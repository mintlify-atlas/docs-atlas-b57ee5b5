description: "Understand ewelink-api error codes, constructor validation errors, and how to handle failed API calls gracefully using try/catch."
---

ewelink-api surfaces errors in two ways: constructor exceptions thrown synchronously when credentials are invalid, and error fields returned in the resolved Promise of API methods. Understanding both patterns helps you build resilient automations.

## Constructor errors

The constructor throws a synchronous `Error` if the credentials you provide are invalid or incomplete. This happens before any network request is made.

```javascript
const ewelink = require('ewelink-api');

try {
  const connection = new ewelink({
    email: 'you@example.com',
    // missing password — will throw
  });
} catch (err) {
  console.error(err.message); // 'Invalid credentials provided'
}
```

Valid constructor combinations are:
- `email` + `password`
- `phoneNumber` + `password`
- `at` (access token alone)
- `at` + `apiKey`
- `devicesCache` + `arpTable` (Zeroconf mode)

## API error codes

When an API call fails, the returned object includes an `error` field (numeric code) and a `msg` field (description). A successful call has `status: 'ok'`.

```javascript
const result = await connection.setDevicePowerState('bad-device-id', 'on');

if (result.error) {
  console.error(`Error ${result.error}: ${result.msg}`);
}
```

### eWeLink server error codes

| Code | Message |
|------|---------|
| `400` | Parameter error |
| `401` | Wrong account or password |
| `402` | Email inactivated |
| `403` | Forbidden |
| `404` | Device does not exist |
| `406` | Authentication failed |
| `503` | Service temporarily unavailable or device is offline |

### Library-level errors

These errors are generated by ewelink-api itself, not the eWeLink server.

| Key | Message |
|-----|---------|
| `ch404` | Device channel does not exist |
| `unknown` | An unknown error occurred |
| `noDevices` | No devices found |
| `noPower` | No power usage data found |
| `noSensor` | Can't read sensor data from device |
| `noFirmware` | Can't get model or firmware version |
| `invalidAuth` | Library needs to be initialized using email and password |
| `invalidCredentials` | Invalid credentials provided |
| `invalidPowerState` | Invalid power state. Expecting: "on", "off" or "toggle" |

## Common error scenarios

### Wrong region → 400 or 401

The most common cause of errors is using the wrong region. The default is `us`. If your account is registered in Europe or Asia, you must pass `region: 'eu'` or `region: 'as'`.

```javascript
// Wrong — will return 400/401 if account is in Europe
const connection = new ewelink({ email, password });

// Correct
const connection = new ewelink({ email, password, region: 'eu' });
```

<Warning>
  Always set the `region` parameter explicitly. Use `getRegion()` if you are unsure which region your account belongs to.
</Warning>

### Device offline → 503

If the device is offline, the API returns error code `503`.

```javascript
const state = await connection.getDevicePowerState('1000abcdef');
// { error: 503, msg: 'Service Temporarily Unavailable or Device is offline' }
```

### Invalid power state

Passing a state other than `"on"`, `"off"`, or `"toggle"` throws an error.

```javascript
// Will throw: 'Invalid power state. Expecting: "on", "off" or "toggle"'
await connection.setDevicePowerState('1000abcdef', 'yes');
```

## Defensive error handling

Wrap API calls in `try/catch` and check the returned `error` field for a complete safety net.

```javascript
const ewelink = require('ewelink-api');

async function turnOnDevice(deviceId) {
  let connection;

  try {
    connection = new ewelink({
      email: process.env.EWELINK_EMAIL,
      password: process.env.EWELINK_PASSWORD,
      region: process.env.EWELINK_REGION || 'us',
    });
  } catch (err) {
    console.error('Failed to initialize:', err.message);
    return;
  }

  const result = await connection.setDevicePowerState(deviceId, 'on');

  if (result.error) {
    console.error(`API error ${result.error}: ${result.msg}`);
    return;
  }

  console.log('Device is now:', result.state);
}
```
